## 前言

手册的主要目的是帮助开发人员提高代码质量，减少重复错误。正如交通规则旨在保护安全而非限制自由，软件规范的目的不是破坏创造力，而是通过限制过度的个性化来提高协作效率。

## 1. 编程规约

### 命名风格

1.  **【强制】** 代码中的命名均不能以下划线或美元符号开始，也不能以下划线或美元符号结束。
    - 反例：`_name` / `__name` / `$Object` / `name_` / `name$`
2.  **【强制】** 代码中的命名严禁使用拼音与英文混合的方式，更不允许直接使用中文。
    - 正例：`alibaba` / `taobao` / `youku` / `Hangzhou`（这些拼音作为专有名词是可以接受的）。
3.  **【强制】** 类名使用 `UpperCamelCase` 风格，但以下情形例外：`DO` / `BO` / `DTO` / `VO` / `AO` / `PO` 等。
    - 正例：`MarcoPolo` / `UserDO` / `HtmlDTO` / `XmlService`
4.  **【强制】** 方法名、参数名、成员变量、局部变量都统一使用 `lowerCamelCase` 风格。
5.  **【强制】** 常量命名全部大写，单词间用下划线隔开，力求语义表达完整清楚，不要嫌名字长。
    - 正例：`MAX_STOCK_COUNT`
6.  **【强制】** 抽象类命名使用 `Abstract` 或 `Base` 开头；异常类命名使用 `Exception` 结尾；测试类命名以它要测试的类名开始，以 `Test` 结尾。
7.  **【强制】** 中括号是数组类型的一部分，数组定义如下：`String[] args;`
    - 反例：`String args[];`
8.  **【强制】** POJO 类中布尔类型的变量，都不要加 `is` 前缀，否则部分框架解析会引起序列化错误。
9.  **【强制】** 包名统一使用小写，点分隔符之间有且仅有一个自然语义的英语单词。包名统一使用单数形式，但是类名如果有复数含义，类名可以使用复数形式。
10. **【强制】** 杜绝完全不规范的缩写，避免望文不知义。
    - 反例：`AbsClass` (AbstractClass); `condi` (Condition)
11. **【推荐】** 如果模块、接口、类、方法使用了设计模式，在命名时需体现出具体模式。
    - 正例：`public class OrderFactory;` `public class LoginProxy;`
12. **【推荐】** 接口类中的方法和属性不要加任何修饰符号（public 也不要加），并加上有效的 Javadoc 注释。

### 常量定义

1.  **【强制】** 不允许任何魔法值（即未经预先定义的常量）直接出现在代码中。
2.  **【强制】** 在 `long` 或者 `Long` 赋值时，数值后使用大写的 `L`，不能是小写的 `l`（易与数字 1 混淆）。
3.  **【推荐】** 不要使用一个常量类维护所有常量，按常量功能进行归类，分开维护。

### 代码格式

1.  **【强制】** 大括号的使用约定：左大括号前不换行，左大括号后换行；右大括号前换行，右大括号后换行（除非后面跟着 else 或逗号）。
2.  **【强制】** `if` / `for` / `while` / `switch` / `do` 等保留字与括号之间都必须加空格。
3.  **【强制】** 采用 4 个空格缩进，禁止使用 tab 字符。
4.  **【强制】** 单行字符数限制不超过 120 个，超出需要换行。

### OOP 规约

1.  **【强制】** 所有的覆写方法，必须加 `@Override` 注解。
2.  **【强制】** `Object` 的 `equals` 方法容易抛空指针异常，应使用常量或确定有值的对象来调用 `equals`。
    - 正例：`"test".equals(object);` 或 `java.util.Objects.equals(a, b);`
3.  **【强制】** 所有的相同类型的包装类对象之间值的比较，全部使用 `equals` 方法比较。
    - **注意**：Integer 在 -128 到 127 之间会缓存，超过该范围 `==` 会失效。
4.  **【强制】** 关于基本数据类型与包装数据类型的使用标准：
    - 所有的 POJO 类属性必须使用包装数据类型。
    - RPC 方法的返回值和参数必须使用包装数据类型。
    - **【推荐】** 所有的局部变量使用基本数据类型。
5.  **【强制】** 构造方法里面禁止加入任何业务逻辑，如果有初始化逻辑，请放在 `init` 方法中。
6.  **【强制】** POJO 类必须写 `toString` 方法。

### 集合处理

1.  **【强制】** 只要重写了 `equals`，就必须重写 `hashCode`。
2.  **【强制】** 不要在 `foreach` 循环里进行元素的 `remove`/`add` 操作。`remove` 元素请使用 `Iterator` 方式。
3.  **【推荐】** 集合初始化时，尽量指定集合初始值大小。

## 2. 异常日志

### 异常处理

1.  **【强制】** 不要捕获 Java 标准库中定义的 `RuntimeException`，如 `NullPointerException`。应尽可能通过预检查避免。
2.  **【强制】** 捕获异常是为了处理它，不要捕获了却什么都不处理而抛弃之。
3.  **【强制】** `finally` 块必须对资源对象、流对象进行关闭，有异常也要做 `try-catch`。
4.  **【强制】** 不能在 `finally` 块中使用 `return`。

### 日志规约

1.  **【强制】** 应用中不可直接使用日志系统（Log4j、Logback）中的 API，而应依赖使用日志框架 **SLF4J**。
2.  **【强制】** 对 `TRACE` / `DEBUG` / `INFO` 级别的日志输出，必须使用占位符 `{}` 的方式，避免字符串拼接。

## 3. MySQL 数据库

### 建表规约

1.  **【强制】** 表达“是/否”概念的字段，必须使用 `is_xxx` 命名，数据类型是 `unsigned tinyint`（1 表示是，0 表示否）。
2.  **【强制】** 表名、字段名必须使用小写字母或数字，禁止出现数字开头。
3.  **【强制】** 小数类型为 `decimal`，禁止使用 `float` 和 `double`（存在精度损失）。

### 索引规约

1.  **【强制】** 业务上具有唯一特性的字段，即使是组合字段，也必须建成唯一索引。
2.  **【强制】** 超过三个表禁止 `join`。

## 4. 工程结构

### 应用分层

- **【推荐】** 结构：
  - **Web 层**：请求转发，基础参数校验。
  - **Service 层**：具体的业务逻辑。
  - **Manager 层**：通用业务处理、第三方接口封装。
  - **DAO 层**：数据访问。

### 领域模型 (Layered Models)

- **DO (Data Object)**: 与数据库表结构对应。
- **DTO (Data Transfer Object)**: Service/Manager 向外传输的对象。
- **VO (View Object)**: Web 层返回给终端的对象。

## 5. 安全规约

1.  **【强制】** 隶属于用户个人的页面或者功能必须进行权限控制校验（防止平行越权）。
2.  **【强制】** 用户敏感数据禁止直接展示，必须进行脱敏处理。
3.  **【强制】** 用户输入的 SQL 参数校验，防止 SQL 注入。禁止字符串拼接 SQL。
4.  **【强制】** 在使用平台资源（如短信、支付等）时，必须有防重放限制（频率限制、验证码）。

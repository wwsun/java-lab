# 27-连接池调优：让数据库访问快如疾风

在 Java Web 开发中，**连接池 (Connection Pool)** 是性能调优的最关键环节之一。Spring Boot 3.x 默认集成了 **HikariCP**，它是目前 Java 生态中最快、开销最小的连接池实现。

---

## 1. 为什么要使用连接池？

### 核心心智模型：餐厅排队
1.  **无连接池**: 每次吃饭（执行 SQL）都要重新盖一间餐馆、招募厨师、准备食材，吃完就拆掉。成本极高，响应极慢。
2.  **有连接池**: 预先准备好 10 张桌子（数据库连接）。客人来了直接入座，吃完后桌子不拆，留给下一个客人。

连接池通过**复用已建立的物理连接**，消除了 TCP 握手和数据库身份验证的巨大开销。

---

## 2. HikariCP 核心参数详解

在 `application.yml` 中，我们通常关注以下三个核心维度：

### 2.1 最大并发：`maximum-pool-size`
-   **定义**: 连接池中允许存在的最大连接数。
-   **黄金法则**: 并不是越多越好。过多的连接会导致数据库频繁进行上下文切换（Context Switch），反而降低性能。
-   **计算公式**: `(核心数 * 2) + 1` 是一个经典的起始经验值。

### 2.2 待机维持：`minimum-idle`
-   **定义**: 池中维护的最小空闲连接数。
-   **最佳实践**: 建议将其设置为与 `maximum-pool-size` 相同的值。这被称为“固定大小线程池”，能有效避免连接频繁创建和销毁带来的抖动。

### 2.3 超时容忍：`connection-timeout`
-   **定义**: 当池中所有连接都在忙时，新请求愿意等待的最长时间（毫秒）。
-   **典型值**: `30000` (30秒)。如果高并发下请求积压严重，超过此值会抛出异常，防止雪崩。

## 3. 诊断与排查

在开发阶段，我们开启了 `MyBatis-Plus` 的标准输出日志：

```yaml
mybatis-plus:
  configuration:
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
```

通过控制台日志，您可以清晰地观察到 SQL 的执行耗时以及连接的获取情况。如果发现查询虽然快，但接口整体响应慢，通常意味着**连接池配置过小**导致线程在排队等待连接。

---

## 4. 常见的生产环境陷阱

1.  **连接泄露**: 也就是拿了连接不归还（在现代 Spring Boot + ORM 环境下极少发生，因为框架会自动关闭）。
2.  **网络抖动导致的连接失效**: 我们通过 `connection-test-query: SELECT 1` 来确保每一个取出的连接都是“热的”且可用的。
3.  **连接生存期过长**: 如果连接在池里待得太久，防火墙可能会强行切断。通过 `max-lifetime` 将其设为略短于数据库服务器超时时间，让它自然交替。
